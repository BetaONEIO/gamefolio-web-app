P0-1: Add Application Error Boundaries
Files:

client/src/components/ErrorBoundary.tsx (new file)
client/src/App.tsx
client/src/pages/HomePage.tsx (and other major pages)
What's wrong: No error boundaries exist anywhere in the app. A single component error will crash the entire application with a white screen.

Fix approach:

Create a class-based ErrorBoundary component with componentDidCatch and getDerivedStateFromError
Wrap the root App component in App.tsx
Add error boundaries around major features (video player, upload, profile)
Log errors to console (add Sentry/error tracking integration point for future)
Display user-friendly fallback UI with reload option
Acceptance criteria:

Throwing an error in any component shows fallback UI instead of crashing
Error is logged to console
User can click "Reload" to recover
App continues working after component error is fixed
Gotchas/Risks:

Error boundaries must be class components (React limitation)
Don't catch errors in event handlers (those need try/catch)
Don't over-nest boundaries (performance impact)

P1-3: Optimize Query Cache Invalidation
Files: client/src/hooks/use-clips.ts

What's wrong: Mutations like useLikeClip invalidate all clips queries instead of specific ones, causing unnecessary refetches of entire feeds.

Fix approach:

Change from queryKey: ['/api/clips'] to specific clip: ['/api/clips', clipId]
Use queryClient.setQueryData() for optimistic updates instead of invalidate
Only invalidate list queries if like count affects sort order
Add exact: true flag to invalidate calls when appropriate
Document query key structure in comments
Acceptance criteria:

Liking a clip only refetches that specific clip's data
Feed doesn't scroll to top or re-fetch on like/unlike
Optimistic updates show immediately
Network tab shows fewer requests after like/comment actions
Gotchas/Risks:

Optimistic updates need rollback on error
Query key structure must be consistent across hooks
Cache timing might cause stale data if not careful with invalidation
